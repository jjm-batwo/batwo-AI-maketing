# /plan-validation 워크플로우

구현 계획서의 장기적 실현 가능성을 검증하고, 문제점의 대응방향을 설립하여 계획서를 수정합니다.

**핵심**: 단순 시뮬레이션이 아닌, **6개월~1년 장기 관점의 시뮬레이션** → **대응방향 설립** → **계획서 수정**.

**트리거**: 계획서 작성 완료 시, 검토 요청 시, "시뮬레이션/검증/문제점 체크/보완" 키워드 시.

> 📖 **상세 프로세스 정의**: `.agent/skills/plan-deep-validation/SKILL.md` 참조.
> 이 워크플로우는 실행 가능한 Step 목록이며, 각 Step의 판단 기준과 철학은 스킬 문서에 정의됩니다.

// turbo-all

---

## Step 1: 대상 계획서 확인

```bash
# 현재 열린 파일 또는 사용자가 지정한 계획서 확인
cat docs/plans/PLAN_{feature}.md
```

대상 파일의 Status, Phase 구조, 예상 규모를 파악합니다.

---

## Step 2: 코드 레벨 근본 원인 분석

계획서의 각 구현 항목에 대해 **실제 소스 코드**를 분석합니다.

### 2-1. 관련 엔티티/모델 파악

```bash
# 도메인 엔티티 구조 확인
cat src/domain/entities/{Entity}.ts
# Prisma 모델 확인
grep -A 20 'model {Model}' prisma/schema.prisma
```

### 2-2. 의존성 체인 추적

```bash
# 영향받는 유스케이스 파일 수 파악
grep -rn '{TargetInterface}' src/application/use-cases/
# 챗봇 도구 하드코딩 확인
grep -rn '{하드코딩_의심_문자열}' src/application/tools/
```

### 2-3. DI 컨테이너 + VO/DTO 호환성

```bash
# DI 토큰 수
grep -c 'Symbol.for' src/lib/di/types.ts
# 관련 VO 구조
cat src/domain/value-objects/{VO}.ts
```

### 2-4. 분석 결과 정리

각 문제를 아래 형식으로 정리:

```
### 문제 N: {제목} {⛔|⚠️|ℹ️}
- 코드 근거: {파일:라인} — {코드 발췌}
- 영향 범위: {N}개 파일
- 근본 원인: {1~2문장}
```

---

## Step 3: 외부 검증

발견된 문제별로 **검증된 해결 근거**를 확보합니다.

### 3-1. 검색 수행

```
1. API/SDK 공식 문서 검색 (search_web)
2. npm 패키지 최신 버전 확인 (context7 MCP)
3. 아키텍처 패턴 실무 사례 검색
4. Breaking Changes / Deprecation 타임라인 확인
5. 향후 1년 내 예정된 변경 사항 확인 (API sunset 등)
```

### 3-2. 검증 규칙

- 최소 **3개 이상 독립 소스** 교차 검증
- 날짜 확인: 최신 자료 우선
- 실제 코드 예시 포함된 자료 우선
- 각 해결에 **[출처: ...]** 명시

---

## Step 4: 장기 시뮬레이션 ⭐ 핵심

대응방향을 적용했다고 가정하고 **시간축을 6개월~1년으로 펼쳐서 시뮬레이션**합니다.

### 4-1. 단기 시뮬레이션 (Phase 연쇄)

각 Phase에 대해:
- 이전 Phase 출력물이 이 Phase 입력으로 충분한가?
- 기존 테스트가 이 변경을 감지할 수 있는가?
- DI 컨테이너가 새 의존성을 해결할 수 있는가?
- 기존 API 소비자가 깨지지 않는가?
- DB 마이그레이션이 기존 데이터를 보존하는가?

### 4-2. 장기 시뮬레이션 (6개월~1년 관점) ⭐

전체 구현 완료 시점부터 시간을 앞으로 흘려보며 시뮬레이션:

| 시나리오 | 질문 |
|----------|------|
| **확장성** | 3번째 플랫폼 추가 시 또 전면 리팩터링인가? |
| **스케일업** | 사용자 10배, 캠페인 100배 시 설계가 버티는가? |
| **유지보수성** | 6개월 후 신규 개발자가 이해하고 수정할 수 있는가? |
| **API 생명주기** | 사용 중인 API의 다음 sunset은? 우리 코드 영향은? |
| **토큰/인증** | 토큰 만료, Rotation, 한도가 운영 문제 되는가? |
| **데이터 일관성** | 멀티 플랫폼 데이터 혼재 시 정확성 보장되는가? |
| **운영 비용** | API 호출, DB, 캐시 비용이 감당 가능한가? |
| **보안/규정** | 새로 저장하는 토큰/정보가 규정에 맞는가? |
| **기술 부채** | 대응방향이 새로운 기술 부채를 만드는가? |

### 4-3. 대응방향 설립

발견된 장기 리스크마다:

```
### 장기 리스크 N: {제목}
- 시나리오: {언제, 어떻게 발생하는가}
- 현재 설계 한계: {왜 대응이 안 되는가}
- 대응방향: {코드 수준 해결 전략}
- 계획서 반영: {어느 Phase에 추가/수정}
```

---

## Step 5: 계획서 수정

분석 + 장기 시뮬레이션 결과를 **실제 계획서에 반영**합니다.

### 5-1. 필수 수정 항목

1. `## 🔍 코드 레벨 근본 원인 분석` — Step 2 결과
2. 각 문제에 `#### 검증된 해결 방향` + 근거 — Step 3 결과
3. Phase 로드맵 수정 (장기 시뮬레이션 대응 태스크 삽입) — Step 4 결과
4. `## ⚠️ 리스크 매트릭스 (검증 기반)` — 장기 리스크 포함, 근거 + 대응방향 컬럼
5. **장기 운영 고려사항 섹션** — API sunset, 토큰 갱신, 스케일업 전략
6. vN 변경 요약 테이블 (추측 vs 검증)

### 5-2. 버전 태그

```markdown
**Status**: 📝 Draft vN — 코드 분석 + 외부 검증 + 장기 시뮬레이션 완료
**Updated**: {날짜}
```

---

## Step 6: 수정된 계획서 2차 장기 시뮬레이션 (검증 루프)

수정된 계획서가 **새 문제를 만들지 않는지** 다시 검증합니다.

- 추가된 대응방향이 기존 Phase와 충돌하지 않는가?
- 대응방향 간 순환 의존이 없는가?
- 장기 시뮬레이션 시나리오를 다시 돌려도 문제 없는가?
- 전체 기간/코드량이 현실적인가?

→ 문제 발견 시: Step 5로 돌아가서 재수정
→ 문제 없음: 사용자에게 v1→vN 변경 요약 제시 후 리뷰 요청

---

## Step 7: 확인

```bash
# 계획서가 정상 저장되었는지 확인
head -10 docs/plans/PLAN_{feature}.md
```

사용자에게 최종 보고:
1. 발견된 문제 수 + 심각도 요약
2. 설립된 대응방향 수
3. 계획서 변경 요약 (vN 테이블)
4. 남은 리스크 (완화는 되었으나 제거 불가한 것들)
