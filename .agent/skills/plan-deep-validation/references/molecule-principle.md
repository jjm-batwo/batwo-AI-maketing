# 태스크 크기 적정성 검증 (분자 단위 원칙)

> **근거**: Liu et al. (2023) "Lost in the Middle" — 컨텍스트 중간 정보 30%+ 성능 저하.
> 컨텍스트 윈도우 70% 초과 시 출력 품질이 Low로 급락 (willness.dev, sfailabs.com).
> REPOCOD 벤치마크 — 대규모 리포지토리에서 LLM pass rate < 30%.

각 Phase의 **태스크가 "분자 단위"인지** 검증합니다.
원자(너무 작음)도, 유기체(너무 큼)도 아닌, **독립적으로 테스트 가능한 기능 단위**여야 합니다.

## 수치적 가이드라인

| 지표 | 너무 작음 (원자) | 적정 (분자) | 너무 큼 (유기체) |
|------|:---:|:---:|:---:|
| 수정 파일 수 | 함수 1개만 | **1~3개 파일** | 5개+ 파일 |
| 예상 소요 | 10분 미만 | **1~3시간** | 4시간+ |
| 독립 테스트 | 의미 없음 | **단독 테스트 가능** | 범위 모호 |
| 맥락 설명 | "X 함수 만들어" | **"Y 기능을 위해 Z까지"** | "전체 시스템" |
| 롤백 범위 | revert 의미 없음 | **단일 커밋 revert 가능** | revert → 다른 것 깨짐 |

## 컨텍스트 오염 위험도 판별

| 위험도 | 조건 | 조치 |
|--------|------|------|
| 낮음 | 1~2개 파일, 같은 레이어 내 | 그대로 진행 |
| 중간 | 3~4개 파일, 2개 레이어 걸침 | 중간에 검증 포인트 추가 (`tsc --noEmit`) |
| **높음** | **5개+ 파일, 3개+ 레이어** | **태스크 분할 필수** |

## 분자 단위 체크리스트

하나라도 "아니오"면 분할/병합 필요:

- [ ] AI에게 목적과 범위를 **한 문장**으로 설명할 수 있는가?
- [ ] 이 태스크만 완료하면 **`tsc --noEmit`이 통과**하는가?
- [ ] 결과를 **독립적으로 테스트**할 수 있는가?
- [ ] 다른 태스크와 **파일 수정이 겹치지 않는가**?
- [ ] 완료 후 **의미 있는 커밋 메시지**를 쓸 수 있는가?
