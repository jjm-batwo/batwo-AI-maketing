# Plan Deep Validation (계획서 심층 검증)

## 🎯 핵심 철학

> **단순 시뮬레이션이 아니라, "장기적 관점의 시뮬레이션"이 핵심이다.**

이 스킬의 목적은 계획서를 **6개월~1년 후까지 시간축으로 펼쳐서**:
1. 지금은 보이지 않지만 **미래에 반드시 터질 구조적 문제**를 사전 발견하고
2. 문제의 **근본 원인을 코드 레벨에서 추적**한 뒤
3. **추측이 아닌 검증된 근거(공식 문서, 실무 사례, npm 코드)**로 대응방향을 설립하고
4. 그 대응방향을 반영하여 **계획서 자체를 수정**하는 것이다.

### 장기 시뮬레이션이란?

"장기 시뮬레이션"은 구현 Phase를 순서대로 따라가면서 **다음 질문들을 던지는 것**:

```
Phase 0 완료 → "이 코드가 3개월 후에도 유지보수 가능한가?"
Phase 1 완료 → "Phase 3에서 이 결정이 병목이 되지 않는가?"
Phase 3 완료 → "Phase 5에서 AI가 이 구조를 이해할 수 있는가?"
전체 완료 후 → "6개월 후 3번째 플랫폼(Naver, Kakao) 추가 시 또 다시 전면 리팩터링인가?"
운영 6개월 → "API 버전 sunset, 토큰 만료, 스키마 드리프트 등 운영 리스크는?"
스케일업 → "사용자 10배, 캠페인 100배 시 현재 설계가 버티는가?"
```

**단기 시뮬레이션**: "이 Phase가 완료되면 다음 Phase가 시작 가능한가?" (Phase 연쇄 점검)
**장기 시뮬레이션**: "이 설계가 6개월~1년 후에도 확장성·유지보수성·운영 안정성을 보장하는가?"

→ **둘 다 수행해야 하지만, 장기 시뮬레이션이 이 스킬의 본질이다.**

## 이 스킬을 호출해야 하는 순간

아래 상황 중 하나라도 해당되면 **반드시** 이 프로세스를 실행합니다:

| 트리거 | 설명 |
|--------|------|
| 새 구현 계획 작성 완료 후 | 작성된 계획서의 장기 실현 가능성 검증 |
| 기존 계획서 검토 요청 시 | 장기 관점 문제점 및 보완사항 도출 |
| "시뮬레이션", "검증", "문제점 체크", "보완" 키워드 | 명시적 검증 요청 |
| 대규모 기능 추가 계획 (예상 2주+) | 리스크가 높은 작업 자동 트리거 |
| 외부 시스템 통합 계획 (API, SDK, 3rd party) | 호환성·구조적 충돌 + 장기 운영 검증 |
| 아키텍처 변경/리팩터링 계획 | 변경 후 장기 영향 시뮬레이션 |

## Anti-Pattern

- ❌ "코드 안 보고 계획만으로 판단" → ✅ **반드시 실제 코드 분석 먼저**
- ❌ "추측으로 문제점 작성" → ✅ **검색/문서/논문으로 근거 확보 후 작성**
- ❌ "문제 발견했으니 끝" → ✅ **대응방향 설립 + 실제 코드 샘플 + 계획서 수정까지**
- ❌ "한 번 검증하면 끝" → ✅ **대응방향 적용 후 장기 재시뮬레이션으로 2차 검증**
- ❌ "Phase 연쇄만 확인" → ✅ **6개월~1년 후 관점까지 시뮬레이션 (장기 관점 필수)**
- ❌ "현재 기능만 기준으로 판단" → ✅ **향후 확장 (3번째 플랫폼, 사용자 10배) 시나리오 포함**

## 적용 수준 자동 판별

**모든 계획에 5단계 전체를 적용하지 않습니다.** 규모와 유형에 따라 자동으로 분기합니다.

| 계획 규모 | 예상 기간 | 적용 수준 |
|-----------|----------|----------|
| **소규모** | 1~3일 | Phase 1(간소화) + Phase 3(장기 시나리오 3개만: 확장성, 유지보수, 기술 부채) |
| **중규모** | 1~2주 | Phase 1~4 전체 |
| **대규모** | 3주+ | Phase 1~5 전체 + 2차 루프 |

| 계획 유형 | Phase 1 적용 |
|-----------|-------------|
| **코드 변경 계획** (기능 추가, 리팩터링, API 통합) | Phase 1 전체 (코드 분석) |
| **비코드 계획** (인프라, 배포, 비즈니스 전략, 가격) | Phase 1 건너뜀 → Phase 2부터 |

## 5단계 프로세스

### Phase 1: 코드 레벨 근본 원인 분석 (Code-Level Root Cause Analysis)

> 계획서에 언급된 각 구현 항목에 대해 **실제 소스 코드**를 열어서 영향 범위를 파악합니다.

#### 체크리스트

- [ ] **영향받는 엔티티/모델 파악**: 관련 도메인 엔티티, Prisma 모델의 **실제 필드 구조** 확인
  ```bash
  cat src/domain/entities/{Entity}.ts
  grep -A 20 'model {Model}' prisma/schema.prisma
  ```

- [ ] **의존성 체인 추적**: 해당 포트/인터페이스를 참조하는 모든 파일 찾기
  ```bash
  grep -rn 'ITargetService' src/application/use-cases/ | wc -l
  grep -rn 'ITargetService' src/application/use-cases/
  ```

- [ ] **DI 컨테이너 영향 확인**: 토큰 등록, 바인딩 수, 모듈 구조
  ```bash
  grep 'Symbol.for' src/lib/di/types.ts | wc -l
  grep -A5 'TargetService' src/lib/di/container.ts
  ```

- [ ] **챗봇/AI 도구 하드코딩 확인**: 플랫폼/기능명 문자열 하드코딩 여부
  ```bash
  grep -rn '하드코딩_의심_문자열' src/application/tools/
  ```

- [ ] **DTO/VO 호환성**: 기존 값 객체가 새 요구사항을 수용할 수 있는지
  ```bash
  cat src/domain/value-objects/{VO}.ts
  cat src/application/dto/{feature}/{DTO}.ts
  ```

#### 출력 형식

각 문제에 대해 아래 템플릿으로 정리:

```markdown
### 문제 N: {제목} {심각도 아이콘}

#### 코드 근거
\`\`\`typescript
// {파일 경로} (실제 코드)
{코드 발췌 — 문제가 되는 부분 하이라이트}
\`\`\`

**영향 범위**: {N}개 파일, {N} import
**근본 원인**: {1~2문장}
```

### Phase 2: 외부 검증 (External Verification)

> 발견된 각 문제에 대해 **추측이 아닌 검증된 근거**로 해결 방향을 확보합니다.

#### 필수 검색 항목

| 검색 대상 | 소스 | 목적 |
|-----------|------|------|
| **API/SDK 공식 문서** | Google, GitHub, npm | 실제 API 사용법, 제약 사항, 인증 요구사항 |
| **npm 패키지 최신 버전** | npmjs.com, Context7 | 라이브러리 실제 코드 패턴, TypeScript 타입 |
| **아키텍처 패턴** | 웹 검색 (Medium, dev.to) | Strategy, Factory, Adapter 패턴 실무 사례 |
| **Breaking Changes** | 공식 블로그, changelog | API 버전 sunset, deprecated 기능, **향후 1년 예정된 변경** |
| **업계 사례** | 레딧, HN, 기술 블로그 | 유사한 문제를 해결한 실무 사례 |
| **장기 운영 이슈** | 공식 문서 deprecation 페이지, 릴리스 노트 | 향후 6개월~1년 내 예정된 breaking changes |

#### 검색 수행 규칙

1. **최소 3개 이상의 독립 소스**에서 교차 검증
2. **날짜 확인**: 2024년 이전 자료는 최신 문서로 재확인
3. **실제 코드 예시** 포함된 자료 우선
4. Context7 MCP로 npm 패키지의 **실제 사용법** 확인
5. **향후 변경 예정 사항** 반드시 검색 (API sunset 일정, deprecation 타임라인)

#### 출력 형식

```markdown
#### 검증된 해결 방향

> **근거**: {구체적 출처와 인용}
> **실제 코드 패턴**: {라이브러리/프레임워크의 검증된 패턴}
> **주의사항**: {공식 문서에서 언급된 gotchas}
> **장기 리스크**: {6개월~1년 후 이 방향에서 발생 가능한 문제}

**대응 코드 방향**:
\`\`\`typescript
// 검증된 패턴 기반 코드 예시
\`\`\`
```

### Phase 3: 장기 시뮬레이션 (Long-Term Simulation) ⭐ 핵심

> **이 Phase가 이 스킬의 본질이다.**
> Phase 2의 대응방향을 적용했다고 가정하고, **시간축을 6개월~1년으로 펼쳐서** 미래에 발생할 문제를 사전에 발견합니다.

#### 시뮬레이션 3단계

##### 3-1. 단기 시뮬레이션 (Phase 연쇄 점검)

각 Phase 시작 전 다음을 점검:

| 질문 | No이면 |
|------|--------|
| 이전 Phase의 출력물이 이 Phase의 입력으로 충분한가? | → 누락된 선행 작업 추가 |
| 기존 테스트가 이 변경을 감지할 수 있는가? | → 테스트 보강 필요 |
| DI 컨테이너가 새 의존성을 해결할 수 있는가? | → DI 등록 누락 체크 |
| 기존 API 소비자(프론트엔드)가 깨지지 않는가? | → 하위 호환성 보장 |
| DB 마이그레이션이 기존 데이터를 보존하는가? | → 데이터 마이그레이션 전략 |

##### 3-1.5 태스크 크기 적정성 검증 (분자 단위 원칙)

> **근거**: Liu et al. (2023) "Lost in the Middle" — 컨텍스트 중간 정보 30%+ 성능 저하.
> 컨텍스트 윈도우 70% 초과 시 출력 품질이 Low로 급락 (willness.dev, sfailabs.com).
> REPOCOD 벤치마크 — 대규모 리포지토리에서 LLM pass rate < 30%.

각 Phase의 **태스크가 "분자 단위"인지** 검증합니다.
원자(너무 작음)도, 유기체(너무 큼)도 아닌, **독립적으로 테스트 가능한 기능 단위**여야 합니다.

**수치적 가이드라인**:

| 지표 | 너무 작음 (원자) | 적정 (분자) | 너무 큼 (유기체) |
|------|:---:|:---:|:---:|
| 수정 파일 수 | 함수 1개만 | **1~3개 파일** | 5개+ 파일 |
| 예상 소요 | 10분 미만 | **1~3시간** | 4시간+ |
| 독립 테스트 | 의미 없음 | **단독 테스트 가능** | 범위 모호 |
| 맥락 설명 | "X 함수 만들어" | **"Y 기능을 위해 Z까지"** | "전체 시스템" |
| 롤백 범위 | revert 의미 없음 | **단일 커밋 revert 가능** | revert → 다른 것 깨짐 |

**컨텍스트 오염 위험도 판별**:

| 위험도 | 조건 | 조치 |
|--------|------|------|
| 낮음 | 1~2개 파일, 같은 레이어 내 | 그대로 진행 |
| 중간 | 3~4개 파일, 2개 레이어 걸침 | 중간에 검증 포인트 추가 (`tsc --noEmit`) |
| **높음** | **5개+ 파일, 3개+ 레이어** | **태스크 분할 필수** |

**분자 단위 체크리스트** (하나라도 "아니오"면 분할/병합 필요):
- [ ] AI에게 목적과 범위를 **한 문장**으로 설명할 수 있는가?
- [ ] 이 태스크만 완료하면 **`tsc --noEmit`이 통과**하는가?
- [ ] 결과를 **독립적으로 테스트**할 수 있는가?
- [ ] 다른 태스크와 **파일 수정이 겹치지 않는가**?
- [ ] 완료 후 **의미 있는 커밋 메시지**를 쓸 수 있는가?

##### 3-2. 장기 시뮬레이션 (6개월~1년 관점) — ⭐ 핵심 중의 핵심

**전체 구현이 완료된 시점**부터 시간을 앞으로 흘려보며 아래 시나리오를 각각 시뮬레이션합니다:

| 시나리오 | 질문 | 발견할 수 있는 문제 유형 |
|----------|------|------------------------|
| **확장성** | "3번째 플랫폼 (예: Naver, Kakao, TikTok) 추가 시 또 전면 리팩터링인가?" | 추상화 레벨 부족, 하드코딩, 플랫폼 의존 |
| **스케일업** | "사용자 10배, 캠페인 100배, 동시 API 호출 50배 시 설계가 버티는가?" | N+1 쿼리, 동시성 병목, rate limiting, DB 인덱스 |
| **유지보수성** | "6개월 후 신규 개발자가 이 코드를 이해하고 수정할 수 있는가?" | 복잡한 의존성, 문서 부재, 매직 넘버, 암묵적 규칙 |
| **API 생명주기** | "사용 중인 API의 다음 버전 sunset은 언제인가? 그때 우리 코드가 영향 받는가?" | 외부 API 버전 의존, 반기별 마이그레이션 비용 |
| **토큰/인증** | "토큰 만료, Rotation, 계정 한도 등이 운영 중 문제가 되는가?" | 자동 갱신 실패, 사일런트 에러, 사용자 재인증 UX |
| **데이터 일관성** | "두 플랫폼 데이터가 혼재될 때 대시보드/리포트가 정확한가?" | 데이터 혼합, 지표 불일치, 화폐 단위 혼동 |
| **운영 비용** | "외부 API 호출 수, DB 크기 증가, 캐시 무효화 빈도가 비용에 미치는 영향은?" | 예상치 못한 API 과금, DB 성능 저하, 캐시 히트율 |
| **보안/규정** | "새로 저장하는 토큰, 개인 정보, 결제 정보가 규정(GDPR, CCPA)에 맞는가?" | 토큰 암호화 누락, 로그에 민감 정보 노출 |
| **기술 부채** | "이 대응방향이 새로운 기술 부채를 만드는가? 나중에 또 리팩터링이 필요한가?" | 임시 방편 코드, 향후 마이그레이션 비용 |

##### 3-3. 대응방향 설립 (Countermeasure Strategy)

> 장기 시뮬레이션에서 발견된 **각 문제에 대해 반드시 대응방향을 설립**합니다.

대응방향은 아래 형식으로 정리:

```markdown
### 장기 리스크 N: {제목}

**시나리오**: {언제, 어떤 상황에서 발생하는가}
**현재 설계의 한계**: {왜 현재 계획으로는 대응이 안 되는가}
**대응방향**: {구체적인 해결 전략 — 코드 수준까지}
**계획서 반영**: {어느 Phase에 어떤 태스크를 추가/수정해야 하는가}
**비용**: {추가 소요 시간, 코드량 변화}
```

#### 신규 리스크 매트릭스

```markdown
| 리스크 | 확률 | 영향 | 근거 | 대응방향 | Phase 반영 |
|--------|------|------|------|----------|------------|
```

---

### Phase 4: 계획서 수정 (Plan Revision) — 대응방향 반영

> Phase 1~3의 결과를 **실제로 계획서에 반영하여 수정**합니다.
> **문제 발견 + 대응방향 설립 + 계획서 수정이 하나의 사이클**입니다.

#### 필수 수정 항목

1. **문제점 섹션 추가/갱신**: `## 🔍 코드 레벨 근본 원인 분석` — Phase 1 결과
2. **각 문제에 대응방향 명시**: `#### 검증된 해결 방향` + 근거 + 코드 — Phase 2 결과
3. **Phase 로드맵 수정**:
   - 장기 시뮬레이션에서 발견된 추가 태스크 삽입
   - 기간, 난이도 재조정
   - 품질 게이트 조건 보강
4. **리스크 매트릭스 갱신**: `## ⚠️ 리스크 매트릭스 (검증 기반)` — 장기 리스크 포함, 근거 + 대응방향 컬럼 필수
5. **장기 운영 고려사항 섹션 추가**: API sunset 대비, 토큰 갱신 전략, 스케일업 전략
6. **v1→vN 변경 요약 테이블**: 추측 vs 검증 비교 — 무엇이 바뀌었고 왜 바뀌었는지

#### 계획서 버전 관리

```markdown
**Status**: 📝 Draft vN — {검증 수준 요약}
**Updated**: {날짜} ({변경 사유})
```

#### 수정 후 자가 검증

수정된 계획서를 다시 읽으면서 확인:
- [ ] 모든 문제에 **근거 기반 대응방향**이 있는가? (추측 금지)
- [ ] 대응방향이 **계획서 Phase에 실제로 반영**되었는가? (문서만 있고 실행 안 되면 무의미)
- [ ] 장기 리스크가 **리스크 매트릭스에 포함**되었는가?
- [ ] Phase 기간/난이도가 **현실적으로 재조정**되었는가?

---

### Phase 5: 수정된 계획서 2차 장기 시뮬레이션 (Validation Loop)

> **수정된 계획서가 장기적으로 문제가 없는지 다시 한번 시뮬레이션합니다.**
> Phase 4에서 추가된 대응방향이 **새로운 문제를 만들지 않는지** 확인합니다.

#### 체크리스트

- [ ] 추가된 대응방향이 기존 Phase와 **충돌하지 않는가?**
- [ ] 대응방향 간에 **순환 의존이 없는가?**
- [ ] 장기 시뮬레이션 시나리오를 다시 돌렸을 때 **새로운 문제가 없는가?**
- [ ] 전체 예상 기간과 코드량이 **현실적인가?**

→ 새 문제 발견 시: Phase 4로 돌아가서 계획서 재수정 (루프)
→ 문제 없음: 최종 버전 확정

**⚠️ 루프 상한: 최대 2회 반복 (Phase 4 → 5 → 4 → 5)**
2회 후에도 문제가 남으면 **"잔존 리스크"**로 분류하여 계획서 리스크 매트릭스에 명시합니다. 모든 리스크를 제거할 수는 없으며, 인식하고 관리하는 것이 목적입니다.

## 전체 흐름 요약

```
📋 계획서 작성 완료
      ↓
Phase 1: 코드 레벨 근본 원인 분석
  "실제 코드에서 문제가 뭔가?"
      ↓
Phase 2: 외부 검증
  "이 문제의 검증된 해결법이 뭔가?"
      ↓
Phase 3: 장기 시뮬레이션 ⭐ 핵심
  "6개월~1년 후에도 이 설계가 살아남는가?"
  "확장, 스케일, 유지보수, API 생명주기, 보안 관점에서?"
  → 문제 발견 시 대응방향 설립
      ↓
Phase 4: 계획서 수정
  "대응방향을 계획서에 실제로 반영"
  → 문제점 + 대응방향 + Phase 수정 + 리스크 매트릭스
      ↓
Phase 5: 수정된 계획서 2차 장기 시뮬레이션
  "수정이 새 문제를 만들지 않는가?"
  → 문제 있으면 Phase 4로 루프
  → 문제 없으면 최종 확정
      ↓
✅ 검증 완료 — 구현 시작 가능
```
